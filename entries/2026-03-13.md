# 2026-03-13

## What I explored
Set up OpenClaw — a local AI agent gateway — from scratch. Configured OpenRouter as the model provider, connected a Telegram bot, created a Family agent using free-tier models, and worked through the Telegram pairing authentication flow.

## Key learnings
- OpenClaw is installed via npm/homebrew and runs a local WebSocket gateway (default port 18789)
- Config lives at `~/.openclaw/openclaw.json` — validated by Zod, hot-reloads when the gateway is running
- **API credentials are NOT stored in `openclaw.json`** — they go in `~/.openclaw/agents/<agent-id>/auth-profiles.json` with `{ "type": "api_key", "key": "..." }` (not `mode`/`apiKey`)
- OpenRouter models are referenced as `openrouter/<author>/<model>` — `openrouter/openrouter/free` routes automatically to the best available free model (zero cost)
- Telegram channel is configured under `channels.telegram.botToken` — `dmPolicy: "pairing"` means unknown senders get a one-time code
- The Telegram pairing flow: user DMs the bot → bot sends a code → user sends it back → gateway shows a pending request → owner runs `openclaw pairing approve <CODE>` in a second terminal
- Pairing requests are held in memory — if the gateway is killed before you approve, the request is lost and you need to DM the bot again for a new code
- `openclaw pairing list` shows pending requests with the exact code — useful when you misread the code
- The `bindings` array in `openclaw.json` routes Telegram traffic to a specific agent: `[{ "agentId": "family", "match": { "channel": "telegram" } }]`
- `openclaw config set <dot.path> <value>` edits the config non-interactively — useful for scripting setup
- `openclaw config validate` confirms the schema is clean before starting the gateway
- `openclaw status` and `openclaw channels status` are the fastest way to confirm everything is wired up correctly

## Surprises / corrections
- The `openclaw channels add --channel telegram` command requires a running gateway — it can't write config cold; use `openclaw config set` instead
- Pairing codes look similar at a glance — `XGBBW7N2` vs `XQBBW7N2` (G vs Q). `openclaw pairing list` shows the exact code to avoid misreads
- `openclaw onboard --non-interactive` exits with error code 1 if the gateway isn't running, even when the config was written successfully — the output confirms success in the text
- `groupPolicy: "open"` triggers multiple security warnings (any group can add and invoke the bot) — fine for personal/family testing, but lock it down for production use

## Connections to past learning
- OpenClaw's `~/.openclaw/openclaw.json` plays a similar role to `~/.claude.json` — both are the central config file for their respective agent runtimes
- The pairing flow is conceptually similar to OAuth device flow — a code is issued, then approved out-of-band by a trusted party
