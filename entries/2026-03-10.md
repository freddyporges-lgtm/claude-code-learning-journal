# 2026-03-10

## What I explored
First real session with Claude Code — focused on understanding the environment, configuring MCP integrations, building the Jeeves weekly update workflow, and setting up this learning journal. Covered a lot of ground quickly, including discovering the gap between Claude Desktop and Claude Code's capabilities.

## Key learnings
- Claude Code and Claude Desktop are completely separate apps — Desktop's OAuth connectors (Slack, Outlook) cannot be used by Claude Code
- MCP servers in Claude Code are configured in `~/.claude.json` under `projects > [path] > mcpServers`
- Two MCP auth models: OAuth/dynamic client registration (works for Atlassian) vs token-based headers (needed for Slack — `mcp.slack.com` rejects dynamic registration)
- Atlassian MCP tools are immediately available in session as `mcp__atlassian__*` — no extra setup needed
- Microsoft 365 MCP is configured but untested — tools would appear as `mcp__microsoft365__*`
- Custom slash commands live in `~/.claude/commands/<name>.md` and are invoked with `/<name>`
- Persistent cross-session memory lives in `~/.claude/projects/[path]/memory/MEMORY.md` (auto-loaded, truncated after 200 lines)
- Superpowers is a plugin (not MCP) that adds dev-oriented skills — brainstorming, planning, TDD, debugging, code review, git worktrees
- GitHub push requires auth — HTTPS needs a PAT, SSH needs a key; `gh` CLI was not installed
- Sharing a PAT in chat is a security risk — always revoke and regenerate after

## Surprises / corrections
- Claude Desktop's integrations are cloud-session-bound — there's no config file to copy from
- `~/.claude.json` has `mcpServers: {}` under the Desktop config — Desktop uses a completely different connector system
- The Superpowers plugin is for OpenCode.ai originally, adapted for Claude Code — its skills are software-dev focused, not workflow automation
- `/journal` as a custom command can't be triggered via the Skill tool — it's a different mechanism
